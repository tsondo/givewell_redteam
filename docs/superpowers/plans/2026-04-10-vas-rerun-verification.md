# VAS Pipeline Re-Run — Adversarial + Linker + Synthesizer

> **For Claude Code:** This plan involves API spend (~$11–16 expected). Tsondo will activate the API key before this plan runs. This is a verification-focused plan: read sanity checks carefully, stop and report on any failure rather than attempting fixes. This project commits directly to main; do not create branches.

**Goal:** Re-run the VAS adversarial, linker, and synthesizer stages against the unchanged upstream outputs (decomposer, investigators, verifier, quantifier) to produce the canonical post-fix VAS artifacts. Verify the new outputs against the sanity checks documented in `docs/linker_info.md` plus synthesizer-specific checks. Commit the new artifacts only after verification passes. Do not write the conclusions document or send anything to Mikyo as part of this plan — those happen after Tsondo reviews the verification report.

**Architecture:** Single resume invocation: `python -m pipeline.run_pipeline vas --resume-from adversarial`. This loads stages 1–4 from disk (`01-decomposer.json` through `04-quantifier.json`, all unchanged) and re-runs stage 5 (adversarial with judge agent), stage 5b (linker), and stage 6 (synthesizer). The new judge agent adds a third API call per critique to the adversarial stage; the linker is a single Sonnet call; the synthesizer is a single Opus call.

## Background

**Plans 1 and 2 have landed.** Plan 1 added the judge agent and updated the debater prompts. Plan 2 added the linker stage and rewrote the synthesizer to consume judge audits, linker dependencies, and decomposer counts directly rather than generating creative content. Both are committed to main and tested.

**The VAS results currently on disk are pre-fix.** `results/vas/05-adversarial.{json,md}` was generated by the old Challenger-assigns-verdict flow and produced 28/28 surviving strong (100% survival rate, which we now know reflects the structural bias rather than actual debate calibration). `results/vas/06-synthesizer.{json,md}` was regenerated once already during the dedup work but still pre-dates the judge agent. There is no `05b-linker.*` file at all because the linker stage didn't exist when VAS was last run.

**Why VAS specifically:** This is the only intervention we are re-running. Water chlorination, ITN, and SMC stay as-is — their pre-fix artifacts are historical record. VAS gets re-run because (a) it's the most recent intervention and the one we plan to send to Brendan, and (b) it has known issues we want corrected before any external sharing (Finding 1's overconfident threshold claim, the missing Conditional Findings section, the absence of judge calibration data).

**What we expect to learn from the new run:**
- Whether the judge agent produces a meaningfully different verdict distribution than 100% strong
- Which failure modes the judge actually detects in real debates
- Whether any surviving findings depend on unverified claims (the linker test)
- Whether the synthesizer correctly reads counts from upstream rather than generating them

**What's NOT in scope for this plan:**
- Conclusions doc update (Plan 4)
- Sending anything to Mikyo (manual step after Tsondo reviews the verification report)
- Sending anything to Brendan
- Re-running any other intervention
- Any code or prompt changes (if something is broken, stop and report)
- Refactoring anything
